## ⚠️ Important: `.env` files don't hide keys from browser users

This is a **pure static frontend** project (plain HTML/JS with no build step like Vite/Webpack). That means:

- `.env` files are a **server-side / build-tool** concept (Node.js, Vite, Next.js, etc.)
- In a static site, **any value your JavaScript uses must be sent to the browser** — there is no way to truly hide an API key from a determined user who opens DevTools
- Even if you use a build tool to inject env vars, the key ends up in your bundled JS file — the browser still receives it

This is actually **by design for Firebase** — the Firebase client config is meant to be public. Firebase security is enforced via **Firestore Security Rules** (which you already have in `firestore.rules`), not by hiding the key.

### 2. Modifying `firestore.rules` to restrict by Domain
You asked to modify `firestore.rules` to only accept requests from `https://b1tsched.netlify.app/`.

**It is actually impossible to restrict domains inside `firestore.rules`.** 
Firestore security rules evaluate the *context of the authenticated user and the data*, but they do **not** have access to HTTP headers like referrers/origins (domain names).

To restrict where your Firebase can be accessed from, you have to do it through the **Google Cloud Console**, not your code:

#### How to restrict your API Key to Netlify:
1. Go to the [Google Cloud Console Credentials Page](https://console.cloud.google.com/apis/credentials) and select your `b1t-sched` project.
2. Find the API key labeled "Browser key (auto created by Firebase)".
3. Under **Application restrictions**, select **HTTP referrers (web sites)**.
4. Add your Netlify URL: `https://b1tsched.netlify.app/*`
5. Click **Save**.

This allows only your Netlify site to use that specific API Key.
